# 知识型阅读工作台前端开发文档

## 1. 文档目标

本文档用于指导「多格式阅读 + 划线批注 + 笔记编辑 + AI 阅读助手」Web 应用的前端开发。

前端目标不是制作概念设计稿，也不是炫技型 AI 页面，而是实现一个可真实上线、可维护、适合长时间阅读的生产级知识工具界面。

开发优先级：

1. 建立稳定的前端工程结构。
2. 建立克制、统一、可复用的设计系统。
3. 完成核心页面和阅读器布局。
4. 完成文件库、阅读器、批注、笔记、AI 面板的基础交互。
5. 使用 mock 数据先打通 UI 和状态，再对接真实 API。

## 2. 产品定位

产品类型：面向学习、研究、办公和知识管理用户的多格式阅读工作台。

核心能力：

- 上传和管理 PDF、DOCX、Markdown、TXT、图片等文件。
- 在统一阅读器中阅读多种格式文档。
- 对选中文本进行划线、高亮、批注和复制。
- 将划线内容摘录到笔记。
- 编辑 Markdown / 富文本笔记。
- 使用 AI 助手对文档进行总结、解释、翻译和问答。

产品关键词：

- 阅读器
- 文件库
- 批注系统
- 知识摘录
- 笔记编辑
- AI 阅读助手
- 真实工具产品
- 长时间阅读场景

视觉参考：

- Notion 的克制感
- Linear 的信息密度和清晰度
- Readwise Reader 的阅读产品感
- Craft / Dropbox Paper 的文档气质
- Apple Notes 的轻量和亲和力

避免方向：

- Web3 Dashboard
- 过度 SaaS 卡片看板
- 过度玻璃拟态
- 夸张渐变科技风
- AI Landing Page 风格
- Dribbble 概念稿风格

## 3. 技术栈

必须使用：

- Vue 3
- Vite
- TypeScript
- Vue Router
- Pinia
- Naive UI
- Tailwind CSS
- PDF.js
- Tiptap
- Markdown-it
- Axios

实现要求：

- 使用 Vue 3 Composition API。
- 使用 TypeScript 定义 API、store、组件 props 和业务实体类型。
- 使用 Tailwind CSS 管理布局和基础样式。
- Naive UI 用作基础组件库，但需要封装项目自己的组件层，避免默认组件库观感。
- 页面必须具备响应式能力。
- 组件职责单一，阅读器相关逻辑必须拆分。

## 4. 设计原则

### 4.1 真实产品感

界面应该像一个已经上线的成熟阅读工具。

要求：

- 不堆无意义装饰元素。
- 不使用大面积炫彩渐变。
- 不使用夸张玻璃效果。
- 不使用大量发光边框。
- 不使用过度阴影。
- 不为了“高级感”牺牲可读性。

### 4.2 避免 AI 生成感

需要避免：

- 所有模块都是圆角卡片。
- 所有卡片都有同样阴影。
- 大标题、渐变背景、统计卡片堆叠。
- 页面元素间距过于平均。
- 图标很多但没有信息价值。
- 每个区域都套容器。
- 所有按钮都很大、很圆、很亮。
- 视觉层级只靠颜色和阴影区分。

### 4.3 克制使用视觉特效

禁止：

- 过度渐变
- 过度阴影
- 过度玻璃拟态
- 过度发光
- 复杂背景纹理
- 夸张动效
- 霓虹科技风
- 过多彩色标签

允许：

- 轻微阴影
- 轻微边框
- 小范围柔和背景
- 简洁 hover 状态
- 适度圆角
- 细腻分割线
- 自然状态反馈

### 4.4 不做卡片堆叠

页面布局应由内容类型决定：

- 文件列表使用表格或紧凑列表。
- 批注使用文本列表或时间线。
- 阅读区域保持沉浸和干净。
- 设置页使用分组表单。
- 工作台可以有少量信息块，但不能全是统计卡片。

### 4.5 信息层级优先

优先使用以下方式建立层级：

- 字号
- 字重
- 行距
- 留白
- 对齐
- 分割线
- 信息密度
- 区块位置
- 文本优先级

尽量少依赖：

- 大色块
- 强阴影
- 强渐变
- 夸张图标
- 大面积卡片

### 4.6 适合长时间阅读

阅读是产品核心，界面不能抢夺文档注意力。

要求：

- 阅读区域安静。
- 侧边栏轻量。
- 高亮颜色柔和。
- 正文字号和行距舒适。
- 工具栏不遮挡正文过多。
- 深浅主题预留。
- 批注操作高效。

## 5. 视觉风格

推荐风格：「克制型知识工具风」。

关键词：

- Clean
- Calm
- Editorial
- Productive
- Focused
- Structured
- Warm minimal
- Real SaaS product

中文描述：

界面应高级但不炫技，像真正面向知识工作者的阅读工作台。它需要克制、清晰、有秩序、有质感，不能像模板后台，也不能像 AI 概念图。

## 7. 字体与排版

字体建议：

- 中文：系统默认字体、PingFang SC、Microsoft YaHei、Noto Sans SC。
- 英文：Inter、system-ui。
- 代码：JetBrains Mono、SF Mono、Menlo。

字号系统：

| 场景 | 字号 / 行高 |
| --- | --- |
| 页面大标题 | 24px / 32px |
| 页面标题 | 20px / 28px |
| 区块标题 | 16px / 24px |
| 正文 | 14px / 22px |
| 阅读正文 | 16px / 28px |
| 辅助文本 | 13px / 20px |
| 表格文本 | 14px / 22px |
| 标签文本 | 12px / 18px |

排版要求：

- 阅读正文行距必须舒适。
- 标题不要全部过度加粗。
- 不使用超大标题制造虚假高级感。
- 文件列表和批注列表需要有较高信息密度。
- 阅读器页面 UI 文案尽量短。
- 页面空间要有节奏，不要所有区域等距排列。

## 8. 布局系统

整体采用：

```text
左侧导航 + 顶部工具栏 + 主内容区域
```

桌面端：

- 左侧导航宽度：`240px`
- 顶部栏高度：`56px`
- 主内容区域自适应
- 阅读器页面使用三栏布局

阅读器页面：

- 左侧文档导航：`240px`，可折叠。
- 中间阅读区域：自适应，控制最大阅读宽度。
- 右侧批注 / AI 面板：`320px`，可折叠。
- 顶部工具栏：`56px`。

响应式断点：

| 宽度 | 布局策略 |
| --- | --- |
| `>= 1280px` | 完整三栏布局 |
| `1024px - 1279px` | 右侧面板可折叠，默认显示阅读区 |
| `768px - 1023px` | 左侧导航收起为图标栏，右侧面板抽屉化 |
| `< 768px` | 单栏布局，阅读器优先，批注和 AI 使用底部抽屉 |

## 11. 页面设计规范

### 11.1 登录页 `/login`

设计目标：简单、可信、安静，不做营销页。

桌面布局：

- 左侧：产品 Logo、产品名、简短说明。
- 右侧：登录表单。

移动端：

- 只显示表单。

内容：

- 产品 Logo
- 产品名
- 简短说明：专注阅读、划线批注与知识整理
- 邮箱输入
- 密码输入
- 登录按钮
- 注册入口

要求：

- 不使用大渐变背景。
- 不使用玻璃拟态登录框。
- 表单真实、克制、清晰。
- 背景使用纯色或极淡纹理。

### 11.2 注册页 `/register`

设计目标：和登录页保持一致。

内容：

- 用户名
- 邮箱
- 密码
- 确认密码
- 注册按钮
- 返回登录

要求：

- 表单验证状态清晰。
- 错误提示自然。
- 不使用花哨动效。

### 11.3 工作台首页 `/dashboard`

设计目标：让用户快速回到最近阅读和最近处理的内容。

不要做传统后台首页，也不要堆统计卡片。

页面结构：

- 顶部：问候语 + 快速上传按钮。
- 主区域左侧：继续阅读列表。
- 主区域右侧：最近批注 / 最近笔记。
- 下方：最近上传文件。

推荐模块：

1. Continue Reading
   - 显示最近阅读文档。
   - 使用列表形式。
   - 包含文件名、类型、阅读进度、上次打开时间。
   - 点击后继续阅读。

2. Recent Annotations
   - 显示最近划线内容。
   - 文本摘录优先。
   - 显示来源文档和时间。

3. Recent Files
   - 使用表格或紧凑列表。
   - 不做大卡片网格。

视觉要求：

- 可以有少量容器，但不要全卡片化。
- 用分割线和留白区分内容。
- 重点突出继续阅读。
- 不使用仪表盘式大数字统计卡。

### 11.4 文件库页面 `/documents`

设计目标：像真实文件管理器，清晰、高效、可筛选。

布局：

- 左侧：文件类型、文件夹、标签筛选。
- 主区域：文件列表。
- 顶部：搜索、上传、排序、视图切换。

默认展示：

- 使用列表或表格。
- 不默认使用卡片墙。

文件列表字段：

- 文件名
- 类型
- 大小
- 上传时间
- 最近阅读时间
- 批注数量
- 操作

交互：

- 搜索文件。
- 类型筛选。
- 上传文件。
- 重命名。
- 删除。
- 打开阅读器。
- 空状态。
- 上传 loading 状态。

视觉要求：

- 文件名是视觉重点。
- 图标只作为辅助。
- 表格行 hover 自然。
- 批注数量使用轻量 badge。
- 不给每个文件做大卡片。

### 11.5 阅读器页面 `/reader/:documentId`

设计目标：沉浸、清晰、适合长时间阅读。

这是产品最核心页面，优先级最高。

整体布局：

```text
ReaderTopBar
  |
ReaderShell
  |-- left: 文档目录 / 页面缩略图占位 / 文件信息
  |-- center: PDF / Markdown / TXT / DOCX 阅读区域
  |-- right: 批注 / 摘录 / AI 面板
```

顶部工具栏：

- 返回按钮
- 文件名
- 文件类型
- 搜索
- 缩放
- 阅读模式
- AI 总结
- 更多操作

左侧面板：

- 文档目录。
- 页面缩略图，MVP 可占位。
- 文档信息。

中间阅读区域：

- PDF / Markdown / TXT / DOCX 渲染区域。
- 阅读内容居中。
- 控制最大阅读宽度。
- 背景安静。
- 选中文本后显示浮动工具栏。

右侧面板：

- 批注列表。
- 笔记摘录。
- AI 助手。
- 使用 Tab 切换。
- 可折叠。

阅读区域要求：

- PDF 页面像真实文档阅读器。
- 文档页与背景之间可以有轻微阴影。
- Markdown / TXT 使用舒适排版。
- 阅读正文不能太宽。
- 文本高亮颜色柔和。
- 当前选区工具栏不能遮挡正文过多。

浮动工具栏：

- 黄色高亮
- 蓝色高亮
- 绿色高亮
- 下划线
- 批注
- 摘录
- AI 解释

浮动工具栏要求：

- 小巧。
- 图标清楚。
- 位置贴近选区。
- 不做大弹窗。
- 不使用强阴影。

批注侧边栏：

- 使用文本列表形式。
- 每条批注展示高亮文本、批注内容、颜色、页码、时间。
- 点击批注定位到原文。
- 支持删除。
- 当前选中批注有轻微背景提示。

AI 助手面板：

- 作为阅读辅助工具，而不是聊天娱乐网站。
- 支持总结当前文档。
- 支持解释选中文本。
- 支持翻译选中文本。
- 支持文档问答。
- 回答区简洁可读。
- 不使用大面积渐变 AI 装饰。

阅读器禁止：

- 大面积玻璃拟态。
- 强烈光效。
- 花哨背景。
- 卡片堆叠。
- 复杂 Dashboard 元素。
- 过度动画。

### 11.6 笔记页面 `/notes`

设计目标：方便整理从文档中摘录的内容。

布局：

- 左侧：笔记列表。
- 中间：笔记编辑器。
- 右侧：来源信息，可折叠。

笔记列表：

- 标题。
- 摘要。
- 来源文档。
- 更新时间。

编辑器：

- 使用 Tiptap。
- 支持 Markdown / 富文本输入。
- 工具栏轻量。
- 展示自动保存状态。
- 来源批注可回跳到原文。

视觉要求：

- 像真实笔记产品。
- 不把每条笔记做成大卡片。
- 重点是内容编辑体验。

### 11.7 设置页面 `/settings`

设计目标：清晰、常规、可维护。

内容：

- 用户资料。
- 主题设置。
- 存储空间。
- AI 配置预留。
- 文件上传限制说明。

布局：

- 左侧设置导航。
- 右侧表单区域。

视觉要求：

- 使用分组表单。
- 少量容器即可。
- 不做卡片堆叠。
- 不做概念化设计。

## 12. 组件开发规范

### 12.1 Layout 组件

#### `AppLayout.vue`

职责：

- 提供全局布局框架。
- 管理侧边栏、顶部栏、主内容区域。
- 处理响应式布局切换。

不负责：

- 具体页面业务逻辑。
- API 请求。

#### `Sidebar.vue`

职责：

- 展示主导航。
- 展示当前路由激活状态。
- 支持折叠。
- 移动端可切换为抽屉或图标栏。

#### `HeaderBar.vue`

职责：

- 展示当前页面标题或轻量操作区。
- 承载全局搜索、上传入口、用户菜单等。

### 12.2 Common 组件

#### `EmptyState.vue`

## 16. 阅读器交互流程

### 16.1 选中文本

流程：

1. 用户在阅读区选中文本。
2. `utils/selection.ts` 获取选中文本、选区矩形、所在容器。
3. `SelectionToolbar` 显示在选区附近。
4. 用户选择高亮、下划线、批注、摘录或 AI 解释。
5. 页面调用 store 创建 annotation 或 note。
6. 创建成功后刷新右侧列表和高亮覆盖层。

### 16.2 PDF 高亮

PDF 高亮必须保存百分比坐标。

示例：

```json
{
  "page": 3,
  "xPercent": 0.22,
  "yPercent": 0.41,
  "widthPercent": 0.35,
  "heightPercent": 0.02
}
```

要求：

- PDF 每页容器使用 `position: relative`。
- `AnnotationLayer` 使用 `position: absolute`。
- 保存坐标时使用百分比。
- 渲染时根据页面尺寸换算为像素。
- 缩放后高亮位置不能明显错位。

### 16.3 Markdown / TXT 高亮

MVP 策略：

- 保存 `selectedText`。
- 保存容器 offset。
- 保存简化 `rangeData`。
- 重新打开时根据 `selectedText` 尝试定位。

后续优化：

- 引入文本锚点系统。
- 使用 TextQuoteSelector 和 TextPositionSelector。
- 支持编辑后重新定位。

### 16.4 批注定位

点击右侧批注：

1. 设置 `activeAnnotationId`。
2. 滚动到对应页面或文本位置。
3. 高亮当前批注。
4. 右侧列表当前项显示轻微背景。

## 17. 关键交互规范

### 17.1 文件上传

- 支持拖拽上传。
- 支持点击上传。
- 上传中显示进度。
- 上传失败有明确错误提示。

- 点击文件名或双击行打开阅读器。
- 支持空状态。

- 打开文档时恢复上次阅读位置。
- 缩放状态可见。
- 左右侧边栏可折叠。

### 17.5 AI 面板

- 按钮文案明确。

## 18. 响应式设计

桌面端：

- 完整三栏阅读器。
- 文件库使用列表表格。
- 工作台使用左右分栏。

平板端：

- 左侧导航变窄。
- 阅读器右侧批注面板默认收起。
- AI 面板使用抽屉。

移动端：

- 单栏布局。
- 底部导航或顶部菜单。
- 阅读器优先展示正文。
- 批注列表使用底部弹层。
- 文件库改为紧凑列表。
- 不显示复杂表格。

要求：

- 移动端适配仍保持 Web 产品一致性。
- 不做成移动 App 概念稿。

## 19. Loading、空状态和错误状态

### 19.1 空状态

空状态要真实、有下一步动作。

示例：

- 暂无文件，上传第一份文档开始阅读。
- 暂无批注，选中文本后可以添加高亮和备注。
- 暂无笔记，可以从划线内容生成第一条笔记。

### 19.2 Loading 状态

必须覆盖：

- 文件列表 loading。
- 文档加载 loading。
- 上传 loading。
- AI 返回 loading。
- 笔记保存 loading。

### 19.3 错误状态

必须覆盖：

- 文件上传失败。
- 文件格式不支持。
- 文档加载失败。
- 登录过期。
- 网络异常。
- AI 服务暂不可用。

错误提示要求：

- 说明问题。
- 给出可执行下一步。
- 不暴露无意义技术堆栈。

## 22. 实施建议

前端开发建议按「设计系统优先，核心路径先通」推进：

1. 先建立 `tokens.css`、`base.css`、布局组件和通用状态组件。
2. 再实现 Dashboard、Documents、Reader、Notes、Settings 的静态 UI。
3. 使用 mock 数据完成页面流转和交互状态。
4. 抽离 reader 组件和选区工具方法。
5. 接入后端 API 和 Pinia store。
6. 最后统一响应式、空状态、loading、错误状态和视觉细节。

这样可以先得到一个真实可用的产品界面骨架，再逐步替换 mock 数据和增强复杂交互。
