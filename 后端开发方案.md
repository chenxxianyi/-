# 知识型阅读工作台后端开发方案

## 1. 文档目标

本文档用于指导「知识型阅读工作台」后端开发，基于根目录《开发方案.md》拆分出后端专项实施方案。

后端建设目标：

1. 提供稳定的 FastAPI REST API。
2. 完成用户认证、文件管理、批注、笔记、AI mock 等 MVP 能力。
3. 使用 MySQL 保存结构化数据。
4. 使用 MinIO 保存文件本体。
5. 使用 Redis 和 Celery 预留异步任务能力。
6. 使用 Alembic 管理数据库迁移。
7. 保证所有文档、批注、笔记操作都有用户权限隔离。
8. 通过 Docker Compose 支持本地完整启动。

MVP 阶段优先保证核心业务闭环可运行，高级能力如 OCR、全文检索、复杂 DOCX 排版、真实大模型接入放入后续迭代。

## 3. 后端职责边界

后端负责：

- 用户注册、登录、JWT 鉴权。
- 用户资料读取和更新。
- 文件上传校验。
- 文件上传到 MinIO。
- 文件元信息保存到 MySQL。
- 文件解析和解析状态管理。
- 为前端提供受控文件预览 URL。
- 保存阅读进度。
- 保存、更新、删除批注。
- 保存、更新、删除笔记。
- 从批注生成笔记。
- 提供 AI 阅读助手 mock API。
- 提供统一响应格式和统一错误处理。
- 对所有用户资源进行权限校验。

后端不负责：

- PDF 页面渲染。
- 前端高亮覆盖层绘制。
- 修改 PDF / Word 原文件。
- MVP 阶段真实调用大模型。
- MVP 阶段 OCR 和复杂版面还原。

## 4. 总体架构

```text
Frontend
  |
  | HTTP / JSON / multipart-form
  v
FastAPI
  |
  |-- api endpoints: 请求接收、依赖注入、响应返回
  |-- services: 业务流程、权限校验、事务协调
  |-- repositories: 数据库读写
  |-- schemas: 请求响应校验
  |-- models: SQLAlchemy ORM
  |
  |-- MySQL: 用户、文档、批注、笔记、解析内容
  |-- MinIO: PDF、DOCX、MD、TXT、图片文件本体
  |-- Redis: 缓存 / Celery broker
  |-- Celery: 文档解析异步任务预留
```

分层原则：

- Endpoint 只负责 HTTP 层。
- Service 负责业务逻辑。
- Repository 负责数据库访问。
- Schema 负责请求和响应数据结构。
- Model 负责数据库映射。
- Utility 负责通用工具函数。

## 9. 文件上传和解析

### 9.1 上传限制

默认限制：

- 最大文件大小：50MB。
- 支持类型：`pdf`、`md`、`txt`、`docx`、`png`、`jpg`、`jpeg`。

必须校验：

- 文件扩展名。
- MIME 类型。
- 文件大小。
- 空文件。

### 9.2 上传流程

```text
POST /api/v1/documents/upload
  -> 获取当前用户
  -> 校验文件大小和类型
  -> 生成唯一 storage_path
  -> 临时保存文件
  -> 上传到 MinIO
  -> 调用 FileParseService 解析
  -> 创建 documents 记录
  -> 更新用户 storage_used
  -> 返回文档信息
```

### 9.3 MinIO 存储服务

文件：`app/services/storage_service.py`

方法：

- `ensure_bucket_exists`
- `upload_file`
- `get_presigned_url`
- `delete_file`

要求：

- 文档详情不直接暴露 MinIO 永久地址。
- 预览通过 presigned URL 或后端受控接口获得。
- 文件软删除时 MVP 可不物理删除 MinIO 对象。

### 9.4 文件解析服务

文件：`app/services/file_parse_service.py`

统一入口：

```python
class FileParseService:
    def parse(self, file_type: str, local_file_path: str) -> ParsedResult:
        ...
```

`ParsedResult` 包含：

- `parsed_content`
- `page_count`
- `word_count`
- `outline`
- `metadata`

MVP 解析策略：

- PDF：使用 `pypdf` 获取页数，正文可以为空。
- Markdown：读取原文内容。
- TXT：使用 `charset-normalizer` 识别编码并读取。
- DOCX：优先使用 `mammoth` 转 HTML，失败时使用 `python-docx` 提取纯文本。
- 图片：返回空解析内容，只提供预览 URL。

### 9.5 解析状态

`parsed_status` 建议值：

- `pending`
- `processing`
- `success`
- `failed`

MVP 可以同步解析。后续改为：

```text
上传文件 -> documents.status = pending -> Celery 解析 -> 更新 parsed_status
```

## 10. 权限校验

所有用户资源必须按 `user_id` 隔离。

必须校验：

- 用户只能查看自己的文档。
- 用户只能修改自己的文档。
- 用户只能软删除自己的文档。
- 用户只能查看自己文档下的批注。
- 用户只能修改和删除自己的批注。
- 用户只能查看、修改、删除自己的笔记。
- 从批注生成笔记时，批注必须属于当前用户。
- AI 文档总结和问答必须校验文档属于当前用户。

推荐实现方式：

- Repository 提供 `get_by_id_and_user_id`。
- Service 层统一调用并处理不存在或无权限。
- 不在 endpoint 中散落权限判断。

## 15. Celery 预留方案

MVP 阶段 Celery 可先搭建基础结构，不强制所有解析异步化。

文件：

- `app/tasks/celery_app.py`
- `app/tasks/document_tasks.py`

预留任务：

- `parse_document_task(document_id)`
- `generate_document_summary_task(document_id)`
- `cleanup_deleted_files_task()`

后续演进：

1. 上传文件后立即创建 document，状态为 `pending`。
2. 投递 Celery 解析任务。
3. Worker 下载 MinIO 文件到临时目录。
4. 调用 FileParseService。
5. 更新 `parsed_status`、`parsed_content`、`page_count`、`word_count`。

## 16. Alembic 迁移策略

初始化：

```bash
cd backend
alembic init alembic
```

生成迁移：

```bash
alembic revision --autogenerate -m "init schema"
```

执行迁移：

```bash
alembic upgrade head
```

要求：

- 所有表结构变更必须通过 Alembic。
- 不直接手写数据库结构到 README 让用户手动执行。
- 初始迁移包含 users、documents、annotations、notes。
- 外键、索引、唯一约束需要同步到迁移文件。

建议索引：

- `users.email`
- `documents.user_id`
- `documents.file_type`
- `documents.deleted_at`
- `annotations.user_id`
- `annotations.document_id`
- `annotations.deleted_at`
- `notes.user_id`
- `notes.document_id`
- `notes.deleted_at`

## 20. 安全要求
