# 知识型阅读工作台开发方案

## 1. 项目目标

本项目是一个「多格式文件阅读 + 划线批注 + 笔记编辑 + AI 阅读助手」Web 应用，面向长文档阅读、资料整理、摘录沉淀和 AI 辅助理解场景。

MVP 优先保证以下链路可运行：

1. 用户注册、登录、JWT 鉴权。
2. 上传 PDF、Markdown、TXT、DOCX 文件。
3. 文件进入 MinIO，元信息进入 MySQL。
4. 文件库可搜索、筛选、重命名、软删除。
5. 阅读器可预览 PDF、Markdown、TXT、DOCX。
6. 选中文本后可高亮、批注、复制、摘录到笔记。
7. 批注和笔记可创建、查看、编辑、删除。
8. 阅读进度可保存。
9. AI 面板通过 mock 接口返回总结、解释、翻译和问答结果。
10. Docker Compose 可启动完整开发环境。

高级能力如 DOCX 精准排版、OCR、真实大模型接入、稳定文本锚点、全文检索和多人协作放入后续迭代。

## 2. 技术栈

### 前端

- Vue 3
- Vite
- TypeScript
- Vue Router
- Pinia
- Naive UI
- Tailwind CSS
- PDF.js
- Tiptap
- Markdown-it
- Axios

### 后端

- Python 3.11+
- FastAPI
- SQLAlchemy 2.x
- Alembic
- MySQL 8
- Redis
- MinIO
- JWT
- Pydantic v2
- Uvicorn
- Celery
- Docker Compose

### 核心原则

- 文件本体存储在 MinIO。
- MySQL 只保存结构化数据和解析内容。
- 批注、高亮、划线不写回原文件，统一保存到 `annotations` 表。
- PDF 由前端通过 PDF.js 渲染。
- Markdown、TXT、自建笔记支持真实编辑。
- DOCX MVP 阶段先提供基础 HTML 或文本预览，保留转换服务结构。
- AI 模块先 mock，代码结构预留 OpenAI、Claude、Ollama、本地模型等 provider。

## 3. 总体架构

```text
Browser
  |
  | Vue 3 + Vite + PDF.js + Tiptap
  v
FastAPI Backend
  |
  |-- MySQL: users, documents, annotations, notes
  |-- MinIO: uploaded files
  |-- Redis: cache / Celery broker
  |-- Celery: document parsing async jobs, reserved for later
  |
  v
AI Provider Adapter
  |-- MockProvider
  |-- OpenAIProvider, reserved
  |-- ClaudeProvider, reserved
  |-- OllamaProvider, reserved
```

前端负责用户交互、阅读渲染、选区采集、高亮覆盖层渲染和笔记编辑。

后端负责鉴权、文件存储、文件元信息、解析内容、批注、笔记、AI mock 接口和权限校验。

## 6. 后端模块设计

### API 分层

- Endpoint：只处理请求入参、依赖注入和响应封装。
- Service：处理业务流程和权限逻辑。
- Repository：封装 SQLAlchemy 查询和持久化。
- Schema：使用 Pydantic v2 定义请求、响应和 DTO。
- Model：使用 SQLAlchemy 2.x ORM 建模。

### 统一响应

成功：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

失败：

```json
{
  "code": 40001,
  "message": "错误信息",
  "data": null
}
```

### 鉴权

- 注册时使用 `passlib[bcrypt]` 加密密码。
- 登录成功后签发 JWT。
- `Authorization: Bearer <token>` 访问受保护接口。
- `get_current_user` 统一解析 JWT 并加载当前用户。
- 所有文档、批注、笔记接口必须校验 `user_id`。

### 文件上传流程

```text
前端选择文件
  -> POST /api/v1/documents/upload
  -> 后端校验大小、扩展名、MIME
  -> 生成唯一 object key
  -> 上传 MinIO
  -> 文件解析服务解析内容
  -> 保存 documents 记录
  -> 返回文档信息
```

MVP 文件限制：

- 最大大小：50MB。
- 支持扩展名：pdf、md、txt、docx、png、jpg、jpeg。

### 文件解析服务

`FileParseService.parse(file_type, local_file_path)` 返回：

```python
ParsedResult(
    parsed_content="",
    page_count=0,
    word_count=0,
    outline=[],
    metadata={}
)
```

MVP 策略：

- PDF：使用 `pypdf` 获取页数，正文可先为空。
- Markdown：按 UTF-8 或识别编码读取原文。
- TXT：使用 `charset-normalizer` 识别编码并读取。
- DOCX：优先使用 `mammoth` 转 HTML；失败时使用 `python-docx` 提取段落文本。
- 图片：不解析正文，返回空内容和预览 URL。

### MinIO 存储服务

`StorageService` 提供：

- `ensure_bucket_exists`
- `upload_file`
- `get_presigned_url`
- `delete_file`，MVP 先预留，不随软删除触发物理删除

### 阅读器布局

```text

  |
ReaderShell
  |-- left: 文档目录 / 缩略图占位 / 文件信息
  |-- center: PdfReader / MarkdownReader / TxtReader / DocxReader
  |-- right: AnnotationSidebar / Note excerpt / AiPanel
```

### 选中文本交互

1. 用户在阅读区选中文本。
2. `utils/selection.ts` 获取 `selectedText`、选区矩形和容器信息。
3. `SelectionToolbar` 浮动显示。
4. 用户选择高亮颜色、下划线、批注、复制或摘录。
5. 创建 annotation 后刷新 Pinia store。
6. `AnnotationLayer` 根据 annotations 重新渲染覆盖层。
7. `AnnotationSidebar` 同步展示批注列表。

### PDF 高亮坐标

PDF 页容器必须是 `position: relative`，高亮层为 `position: absolute`。

保存百分比坐标：

```json
{
  "page": 3,
  "xPercent": 0.22,
  "yPercent": 0.41,
  "widthPercent": 0.35,
  "heightPercent": 0.02
}
```

渲染时根据当前页面容器尺寸换算为实际像素，避免缩放错位。

### Markdown / TXT 高亮策略

MVP 阶段保存：

- `selectedText`
- 选区容器信息
- 简化 offset
- rangeData

重新打开时先按 `selectedText` 尝试定位。后续可升级为文本锚点系统，例如 `TextQuoteSelector`、`TextPositionSelector`。

## 13. 开发阶段计划

3. 创建 `backend` FastAPI 项目结构。

5. 编写 `docker-compose.yml`。
6. 准备 `.env.example`、`requirements.txt` 和 README。

交付：

- 前端可启动。
- 后端 `/health` 可访问。
- Docker Compose 可启动 MySQL、Redis、MinIO。

### 第二阶段：用户认证

目标：完成注册、登录、JWT 和前端鉴权。

## 15. 安全和权限

必须实现：

- 密码只保存 bcrypt hash。
- JWT `SECRET_KEY` 从环境变量读取。
- 上传文件校验扩展名和 MIME。
- 上传大小默认限制 50MB。
- MinIO 文件通过 presigned URL 或后端受控 URL 访问。
- 所有文档、批注、笔记读写都带 `current_user` 校验。
- 软删除数据默认不返回。
- 禁止通过 ID 访问其他用户资源。

## 16. 风险和处理方案

### PDF 高亮定位偏移

风险：缩放、页面重排后固定像素坐标失效。

处理：保存相对百分比坐标，渲染时根据页面实际尺寸换算。

### Markdown / TXT 高亮重定位不稳定

风险：文本编辑后旧选区 offset 失效。

处理：MVP 保存 `selectedText + offset`，后续引入文本锚点系统。

### DOCX 还原度不足

风险：复杂 DOCX 样式、表格、图片无法完整还原。

处理：MVP 使用 mammoth 转 HTML，失败时回退纯文本；README 说明限制。

### AI 真实接入差异

风险：不同模型接口参数、流式输出和费用控制不同。

处理：先抽象 `AIProvider`，MVP 使用 mock，后续按 provider 实现适配器。

### 文件解析耗时

风险：大文件解析阻塞上传接口。

处理：MVP 同步解析小文件，后续将解析任务迁移到 Celery。

推荐按「可运行主链路」推进，而不是先追求局部功能完美：
